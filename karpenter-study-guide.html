The Problem Karpenter Solves
Cluster Autoscaler's limitations

Cluster Autoscaler (CA) works by scaling AWS Auto Scaling Groups. This creates a fundamental constraint: you must pre-define node groups with a fixed instance type (or a limited set). When pods are pending, CA just asks the ASG to add one more node of the type you pre-configured.

This leads to three problems: you often over-provision (pick a large enough instance for the worst case), you can't react to workload shape (a memory-intensive job needs a different instance than a CPU-intensive one), and scaling is slow (CA has to evaluate ASG state and wait for the ASG to respond).

What Karpenter does differently

Karpenter watches for Pending pods directly and calls the EC2 API itself — no ASGs involved. It looks at what the pending pod actually needs (CPU, memory, GPU, architecture) and picks the cheapest instance type that fits, from any family. It provisions the node in seconds, not minutes.

When workloads consolidate (pods finish, traffic drops), Karpenter bin-packs the remaining pods onto fewer nodes and terminates the underutilised ones — CA cannot do this proactively.

The Provisioning Loop
01
Pod created, goes Pending
kube-scheduler cannot find a node that satisfies the pod's resource requests, node selectors, tolerations, and affinity rules. Pod enters Pending with reason Unschedulable.
02
Karpenter controller detects the pending pod
Karpenter watches for pods with the Unschedulable condition. It batches pending pods together (default 1-second window) to provision one well-sized node instead of one node per pod.
03
Karpenter evaluates NodePools
For each matching NodePool, Karpenter simulates which instance types could schedule all pending pods. It filters by: requirements (instance family, zone, capacity type), resource limits, and taints/tolerations. Eliminates instance types that don't fit.
04
Best instance type selected
From the compatible set, Karpenter picks the cheapest option that can schedule all batched pods. For Spot, it calls CreateFleet with capacity-optimized-prioritized strategy — requests multiple instance types simultaneously so AWS picks the one with highest available capacity.
05
NodeClaim created, EC2 instance provisioned
Karpenter creates a NodeClaim object (the desired node spec) and calls EC2 RunInstances / CreateFleet. The node bootstraps, installs kubelet, and registers with the Kubernetes API server. Typically ~30–60 seconds total.
06
Node registers, pods scheduled
Once the node is Ready, kube-scheduler places the pending pods. Karpenter marks the NodeClaim as Launched. Karpenter labels the node with the NodePool name and all instance metadata (type, zone, capacity type, etc.).
The Consolidation Loop (runs continuously)
A
Karpenter evaluates all nodes for consolidation
Periodically simulates: "if I moved all pods on node X to other existing nodes, would they fit?" If yes, the node is a consolidation candidate.
B
Disruption budget check
Before evicting anything, Karpenter checks the NodePool's disruption budget (max % of nodes that can be disrupted at once) and PodDisruptionBudgets on the pods. If the budget is exhausted, waits.
C
Cordon, drain, terminate
Node is cordoned (unschedulable). Pods are gracefully evicted (respecting terminationGracePeriodSeconds). Once empty, EC2 instance is terminated. Replacement node may be provisioned if needed for the evicted pods.
Three Things That Can Disrupt a Node
Consolidation
Workloads can fit on fewer nodes. Karpenter migrates pods and terminates the underutilised node. Controlled by consolidationPolicy.
Drift
Node no longer matches its NodePool (e.g. AMI updated, instance requirements changed). Karpenter replaces it with a compliant node. Enabled by default.
Expiry
Node reaches its expireAfter age (e.g. 720h = 30 days). Karpenter replaces it. Ensures OS patches are applied on a schedule without manual drains.
What a NodePool Is
A NodePool is the Karpenter CRD that defines what kinds of nodes Karpenter is allowed to provision. It has two main parts: the template (what every node from this pool should look like) and the pool-level config (limits, disruption policy). You can have multiple NodePools — Karpenter picks the cheapest matching one when provisioning.
Annotated NodePool — Application Workloads
nodepool-application.yamlkarpenter.sh/v1
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: application
spec:
  template:                          # ① node template
    metadata:
      labels:
        role: application
    spec:
      nodeClassRef:                   # ② links to EC2NodeClass
        apiVersion: karpenter.k8s.aws/v1
        kind: EC2NodeClass
        name: application
      requirements:                   # ③ scheduling constraints
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot", "on-demand"] # Spot preferred, OD fallback
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["c", "m", "r"]  # compute, general, memory
        - key: karpenter.k8s.aws/instance-generation
          operator: Gt
          values: ["3"]              # 4th gen or newer
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
      expireAfter: 720h              # ④ 30-day node rotation
      terminationGracePeriod: 48h   # ④ max drain time
  limits:                           # ⑤ hard ceiling
    cpu: "1000"
    memory: 2000Gi
  disruption:                       # ⑥ consolidation policy
    consolidationPolicy: WhenUnderutilized
    consolidateAfter: 5m
    budgets:
      - nodes: "20%"               # max 20% disrupted at once
① template
Everything under template describes what each provisioned node will look like — labels applied to the node, taints, and the scheduling requirements. Labels are used by pod nodeSelectors or affinity rules.
② nodeClassRef
Links to an EC2NodeClass which holds the AWS-specific config (AMI, subnets, security groups, IAM role). NodePool = what to provision. EC2NodeClass = how to provision it on AWS.
③ requirements
Kubernetes label selectors that constrain what Karpenter can provision. In means "must be one of these values." Both pods AND Karpenter contribute requirements — the intersection must be non-empty for a pod to land on this NodePool. Listing c, m, r instance families gives Spot diversification across many instance types.
④ expireAfter
Nodes older than 720h (30 days) are gracefully replaced. This forces OS patch rotation without manual node drains. terminationGracePeriod is the max time Karpenter will wait for pods to drain before force-terminating.
⑤ limits
Hard ceiling on total resources Karpenter will provision for this NodePool. Once hit, pending pods stay Pending — Karpenter will not provision more nodes. Protects against runaway scaling (e.g. infinite loop creating jobs).
⑥ disruption
WhenUnderutilized: consolidate when pods can fit on fewer nodes. consolidateAfter: wait 5 minutes of being underutilized before acting. Budget: never disrupt more than 20% of this pool's nodes at once.
Key Requirements Labels
Karpenter-specific labels
capacity-type
spot or on-demand. List both with Spot first in priority — Karpenter prefers the cheapest.
instance-category
c (compute), m (general), r (memory), t (burstable — avoid for prod). Multiple values = more Spot options.
instance-size
Can restrict to specific sizes: ["large","xlarge","2xlarge"]. Leave open if you want Karpenter to right-size freely.
instance-generation
Gt "3" means gen 4+ only. Newer gens have better price-to-performance. Avoids very old instances.
Standard Kubernetes labels
kubernetes.io/arch
amd64 or arm64. Graviton (arm64) is cheaper — restrict to amd64 only if your images aren't multi-arch.
topology.kubernetes.io/zone
Pin to specific AZs. Usually leave open so Karpenter spreads across zones for availability.
node.kubernetes.io/instance-type
Can explicitly allow/deny specific types: NotIn ["t3.nano","t3.micro"].
What EC2NodeClass Is
EC2NodeClass is the AWS-provider-specific CRD. While NodePool defines scheduling constraints (what pods need), EC2NodeClass defines how to actually launch that node on AWS — which AMI, which subnets, which security groups, which IAM instance profile. One EC2NodeClass can be shared across multiple NodePools.
Annotated EC2NodeClass
ec2nodeclass.yamlkarpenter.k8s.aws/v1
apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: application
spec:
  amiSelectorTerms:             # ① which AMI to use
    - alias: al2023@latest      # always latest AL2023 EKS AMI

  role: KarpenterNodeRole-cluster # ② IAM role for node

  subnetSelectorTerms:          # ③ which subnets to use
    - tags:
        karpenter.sh/discovery: my-cluster

  securityGroupSelectorTerms:   # ④ security groups
    - tags:
        karpenter.sh/discovery: my-cluster

  tags:                          # ⑤ EC2 instance tags
    Name: karpenter-application-node
    Environment: production
    karpenter.sh/discovery: my-cluster

  blockDeviceMappings:           # ⑥ root volume config
    - deviceName: /dev/xvda
      ebs:
        volumeSize: 50Gi
        volumeType: gp3
        encrypted: true
        deleteOnTermination: true
① amiSelectorTerms
alias: al2023@latest automatically uses the latest EKS-optimised AL2023 AMI for your cluster version. Karpenter resolves this at provision time — so new nodes always get patched AMIs. Alternative: pin to a specific AMI ID, or use a tag selector for your golden AMI.
② role
IAM instance profile role assigned to the EC2 node. Must have the KarpenterNodeRole managed policies (AmazonEKSWorkerNodePolicy, AmazonEC2ContainerRegistryReadOnly, AmazonEKS_CNI_Policy). This is separate from IRSA — it's the node-level role, not the pod-level role.
③ subnetSelectorTerms
Karpenter launches nodes into subnets matching these tags. Tag your private subnets with karpenter.sh/discovery: cluster-name in Terraform/the console. Karpenter automatically uses multiple subnets for AZ spreading.
④ securityGroupSelectorTerms
Same tag-based discovery for security groups. Tag your node security group with the discovery tag. Karpenter automatically applies the cluster's node security group which allows communication with the control plane.
⑤ tags
EC2 instance tags applied to every Karpenter-provisioned node. Critical for cost allocation (you can split costs by NodePool/environment in Cost Explorer), for compliance, and for the discovery tag used by subnet/SG selectors on the node itself.
⑥ blockDeviceMappings
Root EBS volume config. gp3 is cheaper and faster than gp2. Set size large enough for container images + logs. deleteOnTermination: true ensures volumes are cleaned up when nodes are terminated by consolidation.
AMI Families
AL2023
Amazon Linux 2023. Recommended for new clusters. Containerd runtime, SELinux, better security baseline. alias: al2023@latest
AL2
Amazon Linux 2. Legacy — will reach end of support. Use AL2023 for new deployments. alias: al2@latest
Bottlerocket
Container-optimised OS from AWS. Immutable, auto-updates. Good for security-conscious environments. No SSH by default.
Custom AMI
Use amiSelectorTerms: [{tags: {my-tag: "my-value"}}] to pin to a golden AMI built by your team.
userData
Bootstrap script injected into the EC2 user data. Karpenter auto-generates the kubelet config — you add extra steps here (e.g. mount EFS, custom sysctl settings).
instanceStorePolicy
RAID0 to use local NVMe SSDs as ephemeral storage. Useful for high-IOPS workloads like Kafka, etcd.
Consolidation Policies
WhenUnderutilized

Karpenter checks: "can all pods on this node fit on other existing nodes?" If yes → evict pods and terminate node. Actively bin-packs your cluster.

Use for: application node pools with stateless, interruptible workloads. Maximum cost savings.

Risk: pods are periodically evicted and rescheduled. Requires PodDisruptionBudgets to be set, or availability can dip during consolidation.

This is what you used for your application NodePool at Blinq.IO.
WhenEmpty

Only terminates a node when it has zero pods (DaemonSet pods don't count). Much safer — no forced evictions, but also less aggressive savings.

Use for: system node pools, nodes running critical workloads, or any pool where pod disruption is unacceptable.

Behaviour: as workloads naturally complete and pods move away, the empty node is terminated. No active bin-packing.

Use this for your system NodePool to prevent kube-system pods from being evicted.
Disruption Budgets
disruption budget examples
disruption:
  consolidationPolicy: WhenUnderutilized
  consolidateAfter: 5m   # wait 5m before acting on underutilized node
  budgets:
    # Default: max 10% of nodes disrupted at any one time
    - nodes: "10%"

    # During business hours (9am–5pm UTC Mon–Fri): allow 0 disruptions
    - schedule: "0 9 * * 1-5"
      duration: 8h
      nodes: "0"

    # Outside business hours: allow up to 5 nodes disrupted at once
    - nodes: "5"        # absolute count, not percentage
Budgets are evaluated in order — first matching budget wins. Schedule budgets let you pause consolidation during peak hours (so you don't evict pods when users are active) and allow aggressive consolidation at night when it's cheaper and lower risk.
Drift

Drift happens when a node's current state no longer matches what its NodePool or EC2NodeClass specifies. Karpenter detects and replaces drifted nodes automatically.

AMI drift
A new EKS AMI was released (al2023@latest points to a newer AMI). Nodes running the old AMI are drifted → replaced with nodes running the new AMI. This is how Karpenter automates OS patching.
Requirements drift
You changed the NodePool requirements (added a new instance family, changed capacity type). Existing nodes that don't match are replaced.
NodeClass drift
You updated the EC2NodeClass (changed subnets, security groups, volume size). All nodes from that class are considered drifted and are gradually replaced.
Node Expiry

expireAfter: 720h in the NodePool template forces node rotation after 30 days. Even if no new AMI is released, this ensures security patches applied to the base OS are picked up on the next bootstrap.

When a node expires: Karpenter provisions a replacement node first, then drains and terminates the old one (with the disruption budget respected). Zero-downtime rotation — pods move to the new node before the old one is gone.

This is much better than "we update nodes manually" or relying on Managed Node Group update events. With expiry, node patching is automatic and continuous.
How Karpenter Requests Spot

When the pending pod's requirements allow Spot capacity, Karpenter calls EC2 CreateFleet with AllocationStrategy: capacity-optimized-prioritized. This means:

Karpenter submits a list of compatible instance types (e.g. c5.xlarge, c5a.xlarge, m5.xlarge, m5a.xlarge, r5.xlarge…). AWS selects the one with the deepest available Spot capacity pool — the instance type least likely to be interrupted in that moment.

This is why you list multiple instance categories (c, m, r) in your NodePool requirements. The more instance types Karpenter can choose from, the better the interruption rate. Restricting to a single instance type defeats Spot diversification.
Interruption Flow (2-minute warning)
01
AWS sends Spot interruption notice
AWS publishes an EC2 Spot Instance Interruption Warning to EventBridge exactly 2 minutes before the instance is reclaimed. Karpenter subscribes to this event via an SQS queue.
02
Karpenter cordons the node immediately
The node is marked unschedulable instantly. No new pods will be scheduled onto it. This happens faster than the 2-minute window starts.
03
Karpenter provisions a replacement node
Karpenter immediately starts provisioning a new node (different instance type) for the pods that will be evicted. It doesn't wait for the old node to terminate first.
04
Pods drained with respect to PDBs
Karpenter evicts pods respecting PodDisruptionBudgets. If a PDB prevents eviction (minimum available pods would be violated), Karpenter waits. Pods move to the new node as it becomes Ready.
05
AWS terminates the Spot instance
At the 2-minute mark, AWS terminates the instance. Because Karpenter already cordoned and drained it, there are no running workloads left. Clean termination.
What You Need For Spot to Work Reliably
PodDisruptionBudget
Set on every Deployment that runs on Spot nodes. Without a PDB, all replicas could be evicted simultaneously during a Spot interruption.

minAvailable: 1 or maxUnavailable: 1.
terminationGracePeriodSeconds
Give pods time to finish in-flight requests before being killed. Set on the Pod spec. 30s is the Kubernetes default — may need to be longer for long-running jobs or connections.
Readiness probes on the replacement
Traffic is only routed to the new pod after its readiness probe passes. Without this, pods on the new node receive requests before they're ready, causing errors during the transition.
Spot vs On-Demand Selection
When both spot and on-demand are in requirements, Karpenter picks Spot first because it's cheaper. It only falls back to On-Demand when no Spot capacity is available for any of the compatible instance types. This is why listing many instance families matters — it significantly reduces the chance all Spot pools are dry simultaneously.

Why Two NodePools?

If everything ran on Spot, a Spot interruption event could evict Karpenter itself (or CoreDNS, or the CNI). Then you'd have no network, no DNS, and Karpenter wouldn't be able to provision replacement nodes. The cluster would be paralysed.

By putting critical infrastructure on On-Demand nodes in the system pool, Spot interruptions only affect application pods — which have PDBs and graceful shutdown configured. The control plane always stays up.

The interviewer will likely ask "how do you make sure Karpenter doesn't evict itself?" — the answer is: system NodePool is On-Demand only, and critical pods have a taint that prevents them landing on the Spot application pool.

Interview Q&A — Your Setup
"How does a pod end up on the system pool vs the application pool?"
System pool nodes have a taint CriticalAddonsOnly: NoSchedule. Only pods with a matching toleration (kube-system components do) can land there. Application pods have no such toleration, so they can only go on application pool nodes. This is the separation mechanism.
"What happens if all Spot capacity is exhausted?"
The application NodePool has both spot and on-demand in its capacity type requirement. Karpenter prefers Spot (cheaper), but falls back to On-Demand automatically if no Spot capacity is available across all compatible instance types. Pods continue to be scheduled — at a higher cost, but without interruption.
"How did Karpenter save money compared to what you had before?"
Before Karpenter, we had static node groups — nodes ran 24/7 regardless of actual pod load. Karpenter's consolidation terminates underutilised nodes (WhenUnderutilized policy on the application pool). Plus Spot adoption for stateless workloads directly reduced EC2 costs by 60–70% for those node hours. And right-sizing: Karpenter picks the cheapest instance type that fits the actual pod requests, instead of us picking one large enough for the worst case.
