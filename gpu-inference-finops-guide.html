Overview — why this exists

Most "I deployed an LLM" portfolio projects stop the moment inference works. This one goes a layer deeper: it applies FinOps discipline to the GPU layer specifically — the resource type most commonly over-provisioned in AI workloads, because by default a Kubernetes pod claims an entire physical GPU exclusively regardless of how much of it the workload actually uses. It also patches a real resume gap (GPU/MLOps exposure) by reusing skills already proven in production — Terraform, Kubernetes autoscaling, Prometheus/Grafana — pointed at a new target.

2 model replicas sharing one physical T4 GPU via time-slicing, memory-budgeted deliberately

One-liner: a GKE inference platform that shares one physical GPU across multiple LLM replicas via NVIDIA time-slicing, autoscales the GPU node pool to zero when idle, and instruments GPU utilization/cost through DCGM + Prometheus + Grafana — the same cost-attribution pattern used for cloud spend, applied to GPU-hours instead.

Honest project status

This is a live build, not a polished retrospective. The GKE cluster, GPU node pool, and time-sharing config (Phases 0–2) are live and verified via Terraform. The vLLM/GPU serving layer (Phase 3) is fully built but blocked on a GCP GPU quota approval delay — see the Quota Gauntlet section below, itself real first-hand material for "tell me about a time infrastructure didn't cooperate."

With RunPod and GCP support both dead-ending on a fixed deadline, the pipeline was proven end-to-end a different way instead: a CPU fallback pool (zero GPU quota needed) running Ollama in place of vLLM, verified with a real inference request returning a real 200 response. The GPU architecture stays exactly as designed and documented, ready to take over the moment quota clears — nothing here is simulated or faked to look finished.

GKE cluster (zonal)

Single-zone (us-central1-a) rather than regional — qualifies for GCP's one-free-zonal-cluster-per-billing-account fee waiver, so the control plane itself costs nothing.

system-pool

Fixed 1-node, non-GPU e2-medium pool for background workloads (CoreDNS, later Prometheus/Grafana) so they never compete for GPU capacity.

gpu-pool

n1-standard-4 + 1× NVIDIA T4, Spot pricing, autoscaling 0→1 node, tainted so only GPU-tolerating pods can land there, with GKE-native time-sharing configured directly on the node pool.

vLLM

OpenAI-compatible inference server; two replicas of Qwen2.5-3B-Instruct, each capped to 40% GPU memory utilization, sharing the one physical GPU via time-slicing.

hclterraform/main.tf — gpu_pool node config
resource "google_container_node_pool" "gpu_pool" {
  autoscaling {
    min_node_count = 0
    max_node_count = 1
  }
  node_config {
    machine_type = "n1-standard-4"
    spot         = true
    guest_accelerator {
      type  = "nvidia-tesla-t4"
      count = 1
      gpu_sharing_config {
        gpu_sharing_strategy       = "TIME_SHARING"
        max_shared_clients_per_gpu = 4
      }
    }
    taint {
      key = "nvidia.com/gpu", value = "present", effect = "NO_SCHEDULE"
    }
  }
}

Design choice — static autoscaling pool, not Node Auto-Provisioning

GKE's closer analog to AWS Karpenter is Node Auto-Provisioning (NAP), which creates entirely new node pool shapes on demand rather than scaling a pre-defined pool. I used a standard node pool with Cluster Autoscaler instead — simpler to reason about and debug on a quota-constrained project, and it still achieves the core story (GPU nodes scale to zero when idle). NAP is a noted stretch goal, not a gap in understanding.

Design choice — least-privilege node identity

Nodes run as a dedicated service account with exactly 4 IAM roles (log writer, metric writer, monitoring viewer, artifact registry reader) instead of GCP's broad default Compute Engine service account — the node can do what the workload needs and nothing else.

03

The GPU sharing mechanic

Kubernetes normally treats a physical GPU as an indivisible unit — one pod claims the whole card exclusively, even using 10% of it. GKE supports two native sharing strategies to fix that:

TIME_SHARING (used here)

  • Software-level round-robin scheduling between pods
  • Works on any NVIDIA GPU type
  • No memory isolation between sharing pods
  • Configured directly as a node pool setting via Terraform

MIG (Multi-Instance GPU)

  • Hardware-level partitioning — true isolated mini-GPUs
  • Only available on A100 / H100 cards
  • Dedicated memory and compute per instance
  • Too expensive for this project's budget

The nuance almost everyone misses

Time-slicing shares GPU compute scheduling, not GPU memory — every sharing pod still draws from the same physical 16GB pool (only MIG actually partitions memory). So the real ceiling on concurrent replicas isn't the max_shared_clients_per_gpu setting (4, here) — it's how many replicas' memory reservations actually fit in 16GB. A 3B model needs roughly 6GB just for weights, so 4 replicas would try to reserve 24GB and OOM. Sized to 2 replicas at --gpu-memory-utilization=0.4 each instead — an honest number, not the scheduling ceiling maxed out for show.

yamlk8s/vllm-deployment.yaml — resource shape
resources:
  limits:
    nvidia.com/gpu: 1   # scheduler places both replicas on the one sliced GPU node
args:
  - --model=Qwen/Qwen2.5-3B-Instruct
  - --gpu-memory-utilization=0.4  # 2 replicas × 0.4 ≈ 0.8, headroom for KV cache
  - --max-model-len=4096

The quota gauntlet

Getting a single Spot GPU VM to actually launch on a new GCP billing account meant clearing three independent quota dimensions — each failure surfaced as the same generic pod-level error, and the real reason only showed up one layer deeper, in the Cluster Autoscaler's own status:

1
Free-trial accounts can't request GPU quota at all A hard policy restriction, not a UI gap. Fixed by "activating" the billing account to a standard paid account (free credit stays intact).
2
PREEMPTIBLE_CPUS regional quota — limit 0 The vCPU quota specifically for Spot/Preemptible instances, entirely separate from GPU quota. The Spot node needs 4 vCPUs for the instance itself, regardless of the GPU attached. Requested increase to 8 — approved instantly.
3
GPUS_ALL_REGIONS — a global, project-wide GPU cap — limit 0 Separate again from the per-type regional quota (PREEMPTIBLE_NVIDIA_T4_GPUS, already fine at 1). Requested increase to 2 — auto-rejected: "unable to grant additional quota... new project... wait 48h... or contact Sales." A known automated fraud-prevention heuristic on low-history billing accounts.

How the real error was actually found

The pod event just said 0/1 nodes are available: Insufficient nvidia.com/gpu — generic, unhelpful. The precise root cause only surfaced by checking kubectl get configmap cluster-autoscaler-status -n kube-system -o yaml, which carries the Cluster Autoscaler's actual backoff reason per node group, including the exact quota metric name and limit. Worth knowing as a debugging habit: when a scale-up silently fails on GKE, the pod event is not the source of truth.

Debugging war stories

Real issues hit while building this, in order — genuine STAR-story material, not hypotheticals. (The quota gauntlet above is really war story #1, told separately since it deserves the room.)

1
Terraform couldn't authenticate at all gcloud auth login authenticates the CLI; Terraform (and most SDKs) need a separate credential store — Application Default Credentials, set up via gcloud auth application-default login. Two different auth layers for the same account.
2
APIs must be explicitly enabled per project before use Hit twice — Compute Engine API (before GPU quota rows even appear in the console) and Kubernetes Engine API (blocked the first terraform apply with a 403 SERVICE_DISABLED). Fixed both with gcloud services enable.
3
kubectl couldn't reach the cluster after get-credentials Needed a separate gke-gcloud-auth-plugin binary. Even after installing it via gcloud components install, its directory wasn't on PATH — fixed with a symlink into the same directory gcloud itself already resolves from.
4
Changing GPU sharing config forced a node pool replacement gpu_sharing_config can't be updated on a live node pool — some cloud resource attributes are immutable post-creation, so Terraform's plan showed "must be replaced." Safe here since the pool had 0 nodes at the time; would have meant real disruption on a live pool.
5
Original model choice (Llama-3.2-3B) was gated Meta's Llama models require Hugging Face license acceptance and an auth token just to download weights — pure friction with no payoff for a demo. Swapped to Qwen2.5-3B-Instruct, same size class, fully open.
6
Assumed manual NVIDIA device plugin install — GKE does it natively On self-managed Kubernetes (EKS) you hand-install the device plugin DaemonSet and a time-slicing ConfigMap. GKE auto-installs the driver + device plugin the moment a node pool has a GPU attached, and time-sharing is a first-class Terraform node pool setting — simpler than the generic approach, worth knowing the platform difference explicitly.
7
GPU quota permanently blocked against a fixed deadline — pivoted instead of waiting With the global quota request rejected, a self-serve GPU rental (RunPod) blocked on requiring a credit card, and GCP's support chat gated behind a business-domain field that didn't fit, continuing to wait on any of the three wasn't an option. Added a second, GPU-free node pool and swapped vLLM for Ollama (CPU-friendly, official image) purely to prove the surrounding platform — Terraform, GKE, autoscaling, Deployment/Service — end-to-end, while leaving the GPU-specific design fully intact and ready to take over the moment quota clears.
07

How I'd productionize this

Node Auto-Provisioning

Swap the static autoscaling pool for GKE NAP (or Karpenter on AWS) to provision node shapes dynamically instead of scaling a pre-defined pool.

MIG for real isolation

On A100/H100 hardware, use MIG instead of time-slicing where workloads need guaranteed memory isolation from noisy neighbors.

Multi-tenant chargeback

Label pods by team/model and roll GPU-hour cost up per label, the same cost-attribution pattern used for cloud spend tagging.

Idle-GPU alerting

Alertmanager rule on sustained low DCGM_FI_DEV_GPU_UTIL — the GPU equivalent of flagging an idle EC2 instance.

Multi-cloud portability

Hitting GCP's new-account quota wall (see Quota Gauntlet) was a real lesson in not hard-depending on one cloud's provisioning timeline when a deadline is fixed — evaluated a self-serve GPU rental (RunPod) as a fallback path.

Mock Interview Q&A

Click to reveal a strong answer — read it once, then close it and try to say it in your own words.

GPU capacity is the most expensive, most commonly over-provisioned resource in AI infrastructure — Kubernetes hands one pod an entire physical GPU by default, even if it only uses a fraction of it. This project shares one physical T4 GPU across multiple LLM replicas using NVIDIA time-slicing on GKE, autoscales the GPU node pool to zero when idle, and measures the cost/utilization difference against one-GPU-per-replica — the same cost-attribution instinct I used to cut a production AWS bill 50%, applied to GPU-hours.

Training adjusts a model's weights via backpropagation over large datasets — extremely compute-intensive, usually multi-GPU/multi-node. Inference just runs an already-trained model to generate outputs — much lighter, can run on a single GPU. This project is 100% inference; no training happens anywhere in it, which is exactly why a single Spot T4 is enough hardware.

It's a software-level scheduling mechanism, configured through the GPU's device plugin, that lets multiple pods take turns running on one physical GPU — similar to CPU time-sharing. The limitation: it shares compute scheduling, not memory. All sharing pods still draw from the same physical memory pool, so you have to size each workload's memory reservation so N replicas actually fit — the scheduling ceiling you configure isn't automatically the real ceiling.

Follow-up they might ask: "How is that different from MIG?" → MIG partitions the hardware itself into isolated instances with dedicated memory, but only on A100/H100 cards — time-slicing works on any NVIDIA GPU but gives no isolation.

Getting a single GPU VM to actually launch on a new GCP billing account. It required clearing three independent quota dimensions — a regional per-GPU-type quota, a regional Spot-CPU quota, and a global all-regions GPU cap — and each failure surfaced through the exact same generic pod-scheduling error. The real reason only appeared by checking the Cluster Autoscaler's own status configmap, not the pod event. The global quota request was ultimately auto-rejected as a new-account fraud-prevention measure, forcing a real-time decision about fallback infrastructure under a hard deadline.

I used a standard node pool with Cluster Autoscaler (0→1 nodes) instead of NAP. NAP creates entirely new node pool shapes on demand rather than scaling a pre-defined pool — closer to Karpenter's model — but adds real configuration complexity around cluster-wide resource limits and autoprovisioning defaults. On a quota-constrained project, I chose the simpler, more debuggable option that still delivers the core story: GPU nodes scale to zero when idle. NAP is a noted next step, not a knowledge gap.

No AWS credits remaining, and GCP's free trial covered the GPU cost — but free-trial accounts are blocked from requesting GPU quota by policy, so I had to activate a full billing account first, which kept the credit but removed the restriction. Worth noting this isn't GCP-specific: AWS gates GPU instance families behind similar Service Quota approval for new accounts too — the friction is standard fraud-prevention across major clouds, not a GCP weakness.

Some resource attributes can't be updated on a live resource because the cloud provider's API doesn't support an in-place update for that field — Terraform's only option is to destroy and recreate it. The plan output flags this explicitly so you can catch it before applying; it matters most when the resource has live traffic, since a forced replacement there means real disruption, not just a config tweak. I hit this changing GPU time-sharing config — safe in this case since the pool had zero nodes at the time.

In rough priority order: move to Node Auto-Provisioning for dynamic node shapes, use MIG instead of time-slicing on hardware that needs true memory isolation, add per-team/per-model chargeback using pod labels, wire an Alertmanager rule on sustained low GPU utilization as the GPU equivalent of an idle-instance alert, and avoid hard-depending on a single cloud's GPU provisioning timeline when a project has a fixed deadline — evaluate a self-serve GPU rental provider as a fallback path from day one, not as a last-minute scramble.
