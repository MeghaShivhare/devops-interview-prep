01

What is GitOps?

GitOps is an operational framework where Git is the single source of truth for both application code and infrastructure configuration. Instead of running scripts or clicking buttons to deploy, you declare the desired state of your system in Git — and automated tooling reconciles the actual cluster state to match it continuously.

The term was coined by Weaveworks in 2017. The four principles from OpenGitOps are:

Declarative

You describe what you want, not how to get there. YAML files define the desired state. The system figures out the steps.

Versioned & Immutable

Desired state is stored in Git. Every change is a commit. You have full audit history, and rollback is just reverting a commit.

Pulled Automatically

Software agents running inside the cluster pull the desired state from Git and apply it — the cluster reaches out, not the CI system.

Continuously Reconciled

Agents constantly compare actual vs desired state and correct any drift — even if someone manually changed something in the cluster.

Push-based vs Pull-based CD

The key distinction GitOps introduces is pull-based deployment. Traditional CI/CD pipelines push changes — they reach into your cluster from outside. GitOps agents pull.

Push-based (Traditional CI/CD)

  • CI pipeline has kubectl/aws credentials
  • Deploy step runs from outside the cluster
  • Manual rollback requires re-running pipeline
  • Drift goes undetected unless re-triggered
  • Harder to audit: what's actually running?

Pull-based (GitOps)

  • Agent runs inside the cluster
  • No external system has cluster credentials
  • Rollback = revert the Git commit
  • Drift detected and auto-corrected continuously
  • Git history = complete deployment audit trail

One-liner for interviews

"GitOps means the cluster state always converges to what's in Git. Humans write YAML and push commits — the cluster reconciles itself. We stop SSHing into nodes and start writing pull requests."

02

ArgoCD

ArgoCD is a Kubernetes-native GitOps tool. It runs as a controller inside your cluster, watches a Git repository, and ensures your Kubernetes workloads always match the YAML stored there. Think of it as the reconciler — it's the engine that turns "Git commit" into "pods running in the cluster."

Application

The core ArgoCD object. Maps a Git path + revision to a cluster destination. Tracks sync status and health.

AppProject

RBAC boundary. Controls which repos, namespaces, and cluster-scoped resource kinds an Application is allowed to touch.

Sync Policy

Automated or manual. Automated = any Git push triggers a sync. Manual = a human must click sync or run argocd app sync.

Self-Heal

When enabled, ArgoCD reverts any manual changes to the cluster. Someone kubectl-edited a deployment? ArgoCD rewrites it from Git within seconds.

Health Checks

ArgoCD knows how to check if a Deployment, Service, Ingress, etc. is actually healthy — not just created. Promotions wait on real health.

Sync Waves

Ordering within a single sync. Resources with wave -1 sync first, then 0, then 1. Used to install CRDs before workloads that need them.

App of Apps Pattern

Rather than manually applying each ArgoCD Application YAML, you create a single root Application that watches a directory of Application manifests. When ArgoCD syncs the root, it discovers and creates all child Applications. The cluster bootstraps itself from one command.

kind: Application
metadata:
  name: blinq-apps
  namespace: argocd
spec:
  source:
    repoURL: https://github.com/blinq-io/blinqV2.git
    path: infra/argocd/applications  # watches this dir recursively
    targetRevision: deploy-dev
    directory:
      recurse: true
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true     # delete resources removed from git
      selfHeal: true  # revert manual kubectl edits

Sync Waves — why order matters

Some resources must exist before others. CRDs must be installed before resources of that type. An Ingress must have nginx running before it can receive traffic. Sync waves let you sequence within a single sync operation.

-1
CRD installers External Secrets Operator, Reloader — must exist before workloads reference their custom resources
0
Core infrastructure Kargo, ingress-nginx, all workload Applications — the main cluster services
1
Cluster-level config ClusterIssuer (cert-manager) — requires cert-manager CRDs from wave 0
2
Ingress & TLS Ingress objects + TLS certificates — requires nginx healthy from wave 0. Without this ordering, ArgoCD blocks indefinitely waiting for Ingress health.

Common gotcha

An ArgoCD Application watches a branch, not a tag. If you push a GitOps commit to the wrong branch, ArgoCD never sees it and the cluster doesn't update. We had a case where the workflow committed to dev instead of deploy-dev — the app appeared healthy but was running an old image.

03

Kargo

ArgoCD solves continuous delivery — keeping a cluster in sync with Git. But it doesn't answer: how does a new image get promoted from dev to staging to production? That gap is what Kargo fills.

Kargo is a continuous promotion platform for Kubernetes. Built by Akuity (the company behind Argo), it sits on top of ArgoCD and manages the lifecycle of deployable artifacts — tracking what's available, what's been verified, and what should move to the next environment.

Warehouse source of truth
Watches an artifact source — an ECR repository or a Git branch — and creates Freight whenever something new appears. A Warehouse for blinq-server polls ECR every minute; when a new sha-abc123 tag appears, it wraps that in a Freight object.
Freight the artifact bundle
An immutable record of a specific artifact at a specific version. A Freight bundles one or more image tags (or git commits) together. It travels through Stages on its way to production, accumulating verification status at each stop. You promote Freight, not images directly.
Stage the environment
Represents an environment (dev, stage, prod) and a promotion template — a series of steps: clone the repo, update YAML, commit, push, then signal ArgoCD to sync. A Stage declares which Warehouses (or upstream Stages) it accepts Freight from.
Promotion the action
A single run of a Stage's promotion template for a specific Freight. Kargo records the outcome, which steps ran, what git commits were made. A promotion either succeeds (Freight is now on that Stage) or fails with a full log.
ProjectConfig promotion policy
Controls whether a Stage auto-promotes from its upstream source, or requires a human to click "promote" in the Kargo UI. Dev stages run autoPromotionEnabled: true. Stage and prod require manual approval.

Promotion template steps

Each Stage defines its own sequence. These are the built-in step types Kargo provides:

git-clone

Clones a specific branch of the repo into a working directory. Always the first step.

yaml-update

Updates a key in a YAML file using a Kargo expression like ${{ imageFrom("...").Tag }}. No templating engine needed.

git-commit

Commits the changes. The commit message is configurable — ours say chore(gitops): promote blinq-server to stage.

git-push

Pushes to the target branch. In our setup: deploy-dev, deploy-stage, or deploy-prod.

argocd-update

Tells ArgoCD to sync a named Application and waits until it reports Healthy. This is the gate — promotion only succeeds if the cluster actually comes up.

http

Generic HTTP step — used to dispatch GitHub Actions workflows (ECR copy, BVT tests) and send Slack alerts on failure.

ArgoCD vs Kargo — the short version

ArgoCD makes sure the cluster matches what's in a Git branch. It's stateless — it just reconciles. Kargo manages the flow of specific artifact versions through your environments. It tracks which image version is on which environment, controls who can promote what, and makes promotion an auditable, repeatable action rather than a manual git edit.

Per-service independent promotion

A key design decision: each service has its own Warehouse and its own stage-level Stage. You can promote just ai-server to stage without touching blinq-server. This matters in practice — a hotfix for one service shouldn't force you to promote an unrelated service that may not be stage-ready.

Design decision to explain

The 5 execution-planner services (executions-service, execution-manager, execution-thread-manager, outbox-poller, execution-worker) share a single values file, so they're in one combined Stage. If they had separate Stages, two Kargo promotions running simultaneously would both try to commit to the same file and create a git conflict. One Stage, one commit = no race conditions.

ECR image copying

The dev and prod clusters use separate ECR accounts. A promotion to stage doesn't just update the values file — it first dispatches a ecr-copy-images.yml GitHub Actions workflow that copies the sha-* image tag from the dev ECR to the stage or prod ECR. Only then does Kargo update the Helm values and push to the deploy branch.

Full stack: what Kargo integrates with

GitHub Actions

CI builds images, runs tests. On push to dev, builds all service images tagged sha-{12-char-commit} and pushes to ECR. Kargo takes over from there.

AWS ECR

Container image registry. Kargo uses IRSA (IAM Roles for Service Accounts) to authenticate the kargo-controller ServiceAccount for polling image tags.

ArgoCD

Kargo's argocd-update step calls the ArgoCD API to trigger a sync and poll until the Application reports Healthy. Kargo doesn't deploy — ArgoCD does.

Helm

ArgoCD Applications use the Helm chart in the repo. Kargo updates the image.tag key in the per-environment values file. ArgoCD renders the chart with those values and applies the result.

External Secrets

Secrets aren't in Git. ESO pulls them from AWS Secrets Manager into Kubernetes Secrets. Combined with Reloader, a secret rotation in AWS triggers a rolling restart automatically.

Slack

Each Stage has an http step that fires on failure and posts to #deployments with the promotion name and which step failed — ArgoCD sync error vs pre-deploy step.

Deploy branches — why separate from main

ArgoCD watches deploy-dev, deploy-stage, and deploy-prod — not dev/main. The deploy branches contain only infrastructure state (Helm values files, Kargo state files). Kargo commits to them. Application code lives on dev/main. This separation means a broken application commit can't accidentally push broken infra config into a deploy branch, and vice versa.

yaml infra/kargo/warehouse.yaml — blinq-server Warehouse 
apiVersion: kargo.akuity.io/v1alpha1
kind: Warehouse
metadata:
  name: blinq-server-image
  namespace: blinq
spec:
  interval: 1m         # poll ECR every minute
  subscriptions:
    - image:
        repoURL: 132334799376.dkr.ecr.eu-north-1.amazonaws.com/blinq-server
        imageSelectionStrategy: NewestBuild
        discoveryLimit: 20
        allowTagsRegexes:
          - "^sha-"       # only track immutable sha-* tags, not mutable :latest

Real incident to know

Recorder session pods kept failing with ImagePullBackOff on stage because the cloud-worker and selkies-harness images were never promoted from dev-workers to stage-workers. The app services were deployed — but the worker images they reference at runtime were still pointing at dev ECR tags that didn't exist in stage ECR. App promotions and worker promotions are independent. Always promote workers before (or with) app servers when a session feature changes.

Mock Interview Q&A

These are real questions you should be able to answer cold. Click to reveal a strong answer — read it once, then close it and try to say it in your own words. The goal is internalization, not memorization.

Imagine you have a Google Doc that describes exactly how your kitchen should look — which appliances are plugged in, where everything sits. Now imagine a robot whose only job is to read that doc and make your physical kitchen match it, constantly. If you move the toaster, the robot moves it back. If you update the doc to say "add a coffee machine," the robot adds one.

GitOps is that system for software infrastructure. The "doc" is your Git repository. The "robot" is a tool like ArgoCD running inside your Kubernetes cluster. Whenever you want to change something — update an environment variable, scale up a service, deploy a new version — you edit a file in Git and push a commit. The robot detects the change and makes your cluster match. No one logs into servers. No one runs scripts. The cluster converges to whatever is in Git.

Follow-up they might ask: "What if someone manually changes something in the cluster?" → Self-heal detects the drift and reverts it. The cluster always goes back to Git state.

Push-based: Your CI pipeline (Jenkins, GitHub Actions) has credentials to your cluster. When tests pass, it runs kubectl or helm to apply changes. The change is "pushed" from outside the cluster.

Pull-based: An agent runs inside the cluster. It watches Git. When it sees a change, it pulls the new desired state and applies it from within the cluster. No external system needs cluster credentials.

GitOps uses pull-based. The security advantage is significant: you don't have to give your CI system admin access to your cluster. The auditability advantage is also real — the agent continuously reconciles, so drift gets caught immediately rather than only when the next CI run happens.

Personal context: "At Blinq.IO, CI builds and tags the image, pushes to ECR, and stops. Kargo and ArgoCD take over from there. The cluster credentials never leave the cluster."

ArgoCD is a Kubernetes-native GitOps tool. It's a controller that runs inside your cluster and continuously watches a Git repository. When the desired state in Git diverges from the actual state in the cluster, ArgoCD corrects the cluster.

The problem it solves: without it, "what's actually running in my cluster right now?" is a hard question. You'd have to check deployment logs, ask who ran what script, and hope nothing drifted. With ArgoCD, the answer is always "whatever is in this Git branch, applied with this Helm chart." It makes the cluster auditable and reproducible.

  • Sync status: Synced / OutOfSync — is the cluster matching Git?
  • Health status: Healthy / Degraded — are the workloads actually running?
  • Self-heal: reverts manual cluster edits back to Git state
  • Multi-source: combine a shared Helm chart with per-environment values files

They solve adjacent but distinct problems. ArgoCD is a delivery tool — given a Git branch, make the cluster match it. It's stateless about which version of an image is on which environment.

Kargo is a promotion tool — it tracks specific artifact versions (called Freight) as they move through environments. It knows that blinq-server:sha-abc123 is on dev, has been there for 2 hours, and is a candidate to be promoted to staging. ArgoCD doesn't know or care about that.

Think of it this way: ArgoCD reads Git and applies it to the cluster. Kargo decides what gets written to Git in the first place — it's the controller that manages the lifecycle of artifact promotion. They work together: Kargo edits the Helm values files and pushes to the deploy branch; ArgoCD sees the change and syncs the cluster.

One-liner: "ArgoCD is the executor — it applies what's in Git. Kargo is the gatekeeper — it controls what gets into Git in the first place."

At Blinq.IO, the flow is fully automated through dev and manual from staging onward:

  • Developer merges a PR to the dev branch. GitHub Actions triggers app-images.yml.
  • The workflow builds all service images with Docker Bake, tags them sha-{first 12 chars of commit SHA}, and pushes to ECR.
  • Kargo's blinq-server-image Warehouse polls ECR, detects the new tag, creates a Freight object.
  • The dev-apps Stage has auto-promotion enabled — it immediately starts its promotion template. Clones deploy-dev, updates image.tag in values-dev.yaml using the Kargo expression ${{ imageFrom("...").Tag }}, commits, pushes.
  • ArgoCD sees the new commit on deploy-dev, syncs the main-server-dev Application, rolls out new pods. Kargo waits for healthy confirmation.
  • For staging: an engineer opens promote.blinq.io, selects the Freight, clicks Promote. Kargo runs the stage template: copies the image to stage ECR, updates values-stage.yaml, pushes to deploy-stage, waits for ArgoCD. After that, it dispatches BVT tests via GitHub Actions.
  • For production: same manual process, but targets deploy-prod and the prod ArgoCD cluster.

I work through it layer by layer:

  • kubectl describe pod <name> -n dev — get the exact error. Is it "not found," "unauthorized," or "manifest unknown"?
  • Check if the image tag actually exists in ECR: aws ecr describe-images --repository-name blinq-server --region eu-north-1 --query 'imageDetails[*].imageTags[]'. If it's not there, CI didn't finish — maybe the paths filter skipped the workflow because only infra files changed.
  • Check what tag ArgoCD is trying to deploy: kubectl get deploy main-service -n dev -o jsonpath='{.spec.template.spec.containers[0].image}'
  • Compare to what's in the values file: grep "tag:" infra/helm/charts/server/values-dev.yaml
  • If the tag in the values file doesn't match ECR, either the Kargo promotion didn't run, or the image reference in ECR used a different SHA format. Check recent ArgoCD sync and Kargo promotion history.
  • If the tag exists but pull fails with 401: the node IAM role is missing ECR permissions. Check the IRSA annotation on the node or the EC2 instance profile.
We've seen: CI skipped because only infra/ changed (paths filter), worker images not promoted to stage ECR, and wrong tag format from SHA truncation differences.

First, I distinguish between OutOfSync (cluster differs from Git) and a sync failure (ArgoCD tried to sync and got an error). They look similar but need different fixes.

  • Run argocd app get <app-name> or kubectl describe application <name> -n argocd to see the condition message.
  • If the target namespace doesn't exist: kubectl create namespace <ns>. ArgoCD's CreateNamespace=true should handle this, but the AppProject must include the namespace in destinations.
  • If a CRD is missing (ExternalSecret, etc.): either the CRD installer app hasn't synced yet (check sync waves), or the feature isn't supported — disable it in values.
  • If the AppProject is blocking: the namespace or repo isn't listed in the Project's destinations or sourceRepos. We hit this when adding the kargo namespace — had to add it to infra/argocd/projects/blinq.yaml and force-sync the root app.
  • Force sync manually: argocd app sync <app-name> --insecure

The core reason: separation of concerns between application commits and deployment state. When Kargo writes image.tag: sha-abc123 into a values file and commits, that commit should not be mixed with application code changes. If you watch dev directly, every code commit that happens to not change an image could still trigger an ArgoCD sync. More critically, a broken app commit that fails CI could leave the deploy branch in a half-baked state.

With separate deploy branches, only Kargo writes to them. The commit history is clean — every commit is exactly "promote X to Y at time T." You can read the entire deployment history from git log deploy-stage without noise from code changes.

It also gives you an emergency lever: push directly to deploy-prod with a hotfix values change and bypass the normal promotion flow if needed. That's not possible if the deploy branch is the same as your code branch.

The alternative — one combined "promote all services together" stage — sounds simpler but creates coupling. If ai-server has a bug and fails staging validation, you can't promote blinq-server until that's fixed. One bad service blocks everything.

With per-service stages, you promote independently. A hotfix to orchestrator-server goes through its own promotion path. A risky AI feature can sit on dev while the core server gets promoted. It also means the blast radius of a failed promotion is limited to one service's stage, not the entire environment.

The one exception: the 5 execution-planner services share a single stage because they write to the same values-stage.yaml. If they had separate stages, two promotions running simultaneously would cause a git push conflict on that file. The constraint came from the data model, not the principle.

GitOps enabled the cost work in a few ways — it's the foundation that made the other optimizations safe to do:

  • EKS consolidation: We merged dev and staging workloads into a shared cluster with namespace isolation. This was only safe because ArgoCD enforced AppProject boundaries — dev applications literally cannot touch stage namespaces. Without that isolation guarantee in code, merging environments feels risky.
  • Workload rightsizing: Resource requests in Helm values files. Because all resource configs live in Git with a review history, I could safely reduce requests, see the diff clearly, and revert instantly if something went wrong.
  • Karpenter autoscaling: Replaced fixed node groups. Karpenter itself is deployed and configured via ArgoCD — any node pool change goes through Git review, not a manual EC2 console change.
  • Spot instances: Added Spot instance support to node pools in the Karpenter configs in Git. The change is auditable and reversible.

The broader point: GitOps made infrastructure changes low-risk. When a cost optimization meant changing resource allocation, I could experiment, observe, and revert confidently. That confidence let me be more aggressive with the optimizations.

Correct — we never store secret values in Git. The pattern is: External Secrets Operator (ESO) + AWS Secrets Manager + Reloader.

  • Secrets like MONGO_URI, STRIPE_SECRET_KEY live in AWS Secrets Manager as JSON blobs at paths like blinqV2/prod/main-service/env.
  • ESO, deployed via ArgoCD, runs a ClusterSecretStore that authenticates to Secrets Manager using IRSA — the external-secrets ServiceAccount has an IAM role with read-only access to those paths.
  • An ExternalSecret resource (in the Helm chart) tells ESO which keys to pull and what Kubernetes Secret to create from them. This ExternalSecret definition is in Git — just the key names, not the values.
  • Stakater Reloader watches Kubernetes Secrets. When ESO syncs a new value from Secrets Manager, Reloader detects the hash change and triggers a rolling restart of the Deployment that references it.

So the full rotation flow: update a secret in AWS Secrets Manager → ESO syncs within 1h (or force with an annotation) → Reloader restarts the pod → new pod picks up the new value. Zero manual kubectl operations.

Gotcha to mention: ESO v0.10+ removed the v1beta1 API. Everything must use apiVersion: external-secrets.io/v1. We hit this upgrading ESO and had to update all our ExternalSecret and ClusterSecretStore manifests.

A few real ones worth having ready:

  • Cert-manager HTTP-01 timeout on AWS NLB: When setting up prod, HTTPS certificate issuance kept timing out. The issue was that AWS NLBs have a "hairpin" problem — cert-manager, running inside the VPC, can't reach the NLB's external IP to perform its own self-check. Switching from HTTP-01 to DNS-01 challenge (using Cloudflare's API) fixed it completely.
  • Reloader not restarting pods: Secrets were rotating in Secrets Manager but pods weren't picking up the new values. The issue was watchGlobally: false in Reloader's Helm values. That flag restricts Reloader to watching secrets in kube-system only. Our app secrets are in prod. Setting watchGlobally: true fixed it.
  • ArgoCD OutOfSync on ExternalSecret: ArgoCD kept showing ExternalSecret resources as out-of-sync even after a successful sync. ESO's admission webhook was injecting conversionStrategy: Default, decodingStrategy: None, and metadataPolicy: None into live resources. Since these weren't in our Helm template, ArgoCD always saw a diff. Adding them explicitly to the template resolved the permanent drift.
