01

Overview — why this exists

Most public LLMOps/cost content comes from ML engineers. This project takes the infra/DevOps angle instead: LLM API spend is untracked in most orgs the same way AWS spend was before tagging discipline — nobody notices which team or model is burning budget until the bill arrives. This reuses a skill set I already have (Prometheus/Grafana, cost-attribution thinking from the AWS cost reduction work) and points it at a new target — LLM spend — rather than learning an unrelated stack from scratch.

Architecture

One request's full journey through the stack:

mock-llm

Minimal FastAPI app mimicking OpenAI's chat completions endpoint. Sleeps 0.2–1.2s per request, fakes token usage, randomly returns 500s (~5%). Exists so the stack runs free and offline — swappable for a real provider with a one-line config change.

LiteLLM proxy

Routes requests to whichever backend a model alias maps to, retries transparently on failure, computes per-request cost from configured pricing, and exports everything as Prometheus metrics.

Prometheus

Scrapes LiteLLM's /metrics/ endpoint every 5 seconds and stores the resulting time series, keyed by labels like requested_model and status_code.

Grafana

Fully file-provisioned — one datasource, one 8-panel dashboard, zero manual clicking. Queries Prometheus directly via PromQL on a 10s refresh.

Why a mock backend, not a real provider

Zero API keys, zero real spend, fully offline. Nothing downstream of LiteLLM cares what's actually behind it — swapping mock-llm for real OpenAI/Anthropic/Bedrock is a one-line change to api_base in LiteLLM's config. Nothing else in the pipeline changes.

03

The cost mechanic

mini-model and flagship-model both route to the exact same mock backend — but each carries different configured per-token pricing in LiteLLM's config, modeled after real GPT-4o-mini vs GPT-4o rates:

mini-model

  • $0.00000015 / input token
  • $0.0000006 / output token
  • Modeled after GPT-4o-mini pricing

flagship-model

  • $0.0000025 / input token
  • $0.00001 / output token
  • Modeled after GPT-4o pricing
yaml litellm/config.yaml 
model_list:
  - model_name: mini-model
    litellm_params:
      model: openai/demo-model
      api_base: http://mock-llm:8080/v1
    model_info:
      input_cost_per_token: 0.00000015
      output_cost_per_token: 0.0000006

  - model_name: flagship-model
    litellm_params:
      model: openai/demo-model   # same backend
      api_base: http://mock-llm:8080/v1
    model_info:
      input_cost_per_token: 0.0000025
      output_cost_per_token: 0.00001

The key mechanic

LiteLLM prices a response using the pricing attached to the requested model alias — not whatever backend actually served it. That's what makes a two-tier cost breakdown possible against a single physical backend. In production, the same mechanic runs in the other direction: an alias like internal-summarizer can point at whatever real backend you want without callers ever knowing.

LiteLLM also retries automatically on backend failure. When mock-llm's simulated 500s hit, the client still saw 200s — visible in the metrics as litellm_deployment_total_requests_total (includes retries) running higher than litellm_deployment_success_responses_total (final successes only).

What "p95" actually means

Sort every request's response time slowest to fastest. p95 is the value where 95% of requests were faster than it — only the slowest 5% took longer. It's deliberately not an average: a couple of extreme outliers barely move an average, but they're exactly what makes a real user notice things feel slow. p95 (and its stricter cousin p99) answers "how bad does it get for the unlucky-but-not-rare tail," which is what most latency SLOs are actually written against.

In this stack, the mock backend only sleeps 0.2–1.2s per request, but p95 often reads several seconds higher — because concurrent requests from a traffic burst queue up behind each other (queueing delay counts as latency too), and LiteLLM's automatic retries on the ~5% simulated failures add real wall-clock time. Both are honest contributors, not a measurement bug — see War Story #6.

05

Debugging war stories

Real issues hit while building this, in order — good STAR-story material since they're genuine bugs, not hypotheticals.

1
LiteLLM silent for ~60–90s on every start No log output during startup made it look hung. Confirmed via docker stats that CPU stayed non-zero — it was loading its internal pricing map, not crash-looping.
2
Prometheus target down, HTTP 401 LiteLLM's /metrics requires bearer auth by default; Prometheus's static scrape config sends none. Fixed with require_auth_for_metrics_endpoint: false (acceptable since the endpoint isn't exposed outside the Docker network).
3
Scrape still failing after the auth fix /metrics 307-redirects to /metrics/, and Prometheus doesn't follow redirects on scrape. Pointed metrics_path at the resolved path directly.
4
PromQL dropped a label mid-query Subtracting two metrics with - on (model_id) ... matched correctly but discarded requested_model from the result, collapsing per-tier data into one number. Fixed by aggregating each side down to requested_model before subtracting.
5
Grafana: "data source not found" after a config change docker-compose restart reuses the same container filesystem (incl. Grafana's sqlite state), so a datasource UID change collided with the old provisioned record. Fixed with up -d --force-recreate — a clean container instead of a restarted one.
6
p95 latency read ~9s against a 0.2–1.2s backend Single Uvicorn worker meant concurrent requests queued behind each other — queueing delay counted as latency, not service time. Fixed with 4 workers. Same class of issue as an under-provisioned connection pool inflating p95/p99 without the service actually being slower.
06

How I'd productionize this

Real backend

Swap mock-llm for OpenAI/Anthropic/Bedrock — one line (api_base) in config, nothing else changes.

Long-term storage

Prometheus alone isn't durable/long-retention — add remote-write to Thanos or Mimir.

Alerting

LiteLLM already exposes litellm_remaining_api_key_budget_metric — wire an Alertmanager rule on spend-rate-exceeds-budget, plus error-rate and latency-SLO alerts.

Chargeback

Metrics already carry team, user, and api_key_alias labels — map those directly onto per-team/per-project cost breakdowns.

Metrics auth

Replace the disabled-auth shortcut with a scoped scrape-config credential or network-level isolation (private subnet / security group).

Mock Interview Q&A

Click to reveal a strong answer — read it once, then close it and try to say it in your own words.

Cost and usage visibility for LLM API calls — the same blind spot cloud spend had before tagging discipline existed. Nobody notices which team or model tier is burning budget until the bill arrives; this makes that visible and queryable in real time, using LiteLLM as a proxy in front of any provider, Prometheus to store the metrics, and Grafana to visualize spend/tokens/latency/errors broken down by model tier.

Without it, there are two worse options. Roll your own: every provider (OpenAI, Anthropic, Bedrock) has its own request format, auth style, and pricing table — you'd hand-write "if it's OpenAI, price it this way; if it's Anthropic, price it that way" and maintain it forever as prices change. Pay for a hosted LLM-observability SaaS (Helicone, Langfuse, etc.) — works, but routes your data through a third party and costs money.

What LiteLLM actually buys you: one request/response shape regardless of what's behind it (which is also why a mock backend could stand in for a real provider so easily — it just had to speak that one shape); pricing math it already knows how to do from a one-time price list instead of code you'd write yourself; Prometheus metrics with one config line instead of building a metrics system from scratch; and automatic retries on transient failures for free. It's also self-hosted and free — nothing leaves your own infra.

One-liner: LiteLLM turns "build a billing and metrics system for AI calls" into "write a two-line price list" — that's the entire reason it's in this project instead of a hand-rolled proxy.

LiteLLM prices a response using the model_info pricing tied to the requested model alias, not whatever backend actually served it. Pricing is configuration, fully decoupled from the physical backend — so two aliases pointed at one mock server can still show a completely different cost breakdown in the dashboard.

Follow-up they might ask: "Is that realistic?" → Yes — it's the same mechanic you'd use in reverse in production: point an internal alias at whichever real backend you want, and callers never need to know.

Metric name and label discovery. LiteLLM's Prometheus metric names aren't fully documented in one place, so I verified every single PromQL query against the live /metrics endpoint and Prometheus's query API with real traffic before trusting it in a dashboard panel — otherwise you get a panel that parses fine but silently shows nothing, which is worse than an obvious error because it looks done when it isn't.

In this local stack, yes it's a shortcut — but a scoped one: the endpoint is only reachable inside the Docker network, never exposed to a host port or the internet. In production I wouldn't take that shortcut; I'd either give Prometheus's scrape config a scoped bearer credential or rely on network-level isolation (private subnet, security group) so the metrics endpoint is unreachable regardless of its own auth setting.

This is a good one to bring up proactively — naming the trade-off yourself signals you understand it wasn't an oversight.

Five things, in rough priority order: swap the mock backend for a real provider (one config line), add remote-write to a long-term store like Thanos or Mimir since Prometheus alone isn't durable, wire budget/error-rate/latency alerts through Alertmanager, use the team/user/api_key_alias labels LiteLLM already emits for chargeback, and replace the local auth shortcut on the metrics endpoint with proper scoped credentials or network isolation.
