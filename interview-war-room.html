"Why GitOps over push-based CD?"
Git is the audit trail (who changed what, when, via which PR). Drift is auto-corrected. Cluster credentials never leave the cluster. Manual kubectl changes are overwritten on next sync — nothing can drift silently.
"What did Kargo add that ArgoCD alone couldn't do?"
ArgoCD syncs a declared state. It doesn't know about "promote version X from dev to stage after tests pass." Kargo adds the promotion pipeline layer — tracks artifact versions per stage, runs verification gates, automates the flow while giving visibility into what's where.

CloudWatch — Full Service Study Guide
01 Logs — Architecture
Log Group
Container for log streams from the same application or resource. You set retention policy and storage class here. Example: /aws/eks/blinqV2-prod-cluster/cluster
Log Stream
Sequence of events from a single source within a log group. For EKS, each node or pod gets its own stream. For Lambda, each function instance gets its own stream.
Log Event
A single entry: timestamp + raw message string. CloudWatch indexes by timestamp only — not field-aware (that's what Log Insights adds on top).
Retention Policy
1 day → 10 years, or Never Expire (default). Log groups with no retention policy accumulate forever and bill forever. Set retention on every group you create.

03 Subscription Filters
What it does
Streams matching log events in real-time (near-real-time) to a downstream destination. Pattern match is applied — only matching lines are forwarded.
Destinations
Lambda (real-time processing), Kinesis Data Streams (buffer → consumer), Kinesis Firehose (S3 / Splunk / OpenSearch delivery), Amazon OpenSearch Service (direct).
Limit
Max 2 subscription filters per log group (Standard only). Cross-account subscriptions possible via resource policies.
Use cases
SIEM ingestion, real-time error alerting via Lambda, log centralisation across accounts, shipping logs to Elasticsearch/OpenSearch for full-text search.
Why you checked this before IA conversion: if a subscription filter existed, converting to IA would silently break the stream — no error thrown, logs just stop flowing to the destination.
04 Metric Filters
What it does
Scans incoming log events for a pattern → emits a CloudWatch Metric data point on each match. Bridges logs and metrics. Standard log groups only.
Filter pattern syntax
Simple: ERROR matches any line containing "ERROR". JSON: { $.level = "error" }. Space-delimited: [ip, user, timestamp, method="POST", ...]. Extract values: { $.latencyMs = * } → use as metric value.
Common use
Count ERROR lines per minute → metric → alarm. Extract HTTP status 5xx counts from access logs → metric → alarm. Extract latency from structured JSON logs → p99 alarm.
Limitation
Only processes new log events from the moment the filter is created — does not backfill historical data. Max 2 metric filters per log group in some older accounts.
Why you checked this before IA conversion: if a metric filter existed, converting to IA would silently stop populating the metric — any alarm on it would go INSUFFICIENT_DATA and stop firing.
05 CloudWatch Logs Insights
What it does
Interactive query engine for log data. Works on both Standard and IA log groups. Queries scan log data on demand — you pay per GB scanned.
Key commands
fields @timestamp, @message — select fields
filter @message like /ERROR/ — filter by pattern
filter @message like /ERROR/ | stats count(*) by bin(5m) — error rate over time
parse @message "* * *" as ip, user, action — extract fields from unstructured logs
sort @timestamp desc | limit 50 — most recent first
Auto-discovered fields
For JSON logs, CloudWatch auto-discovers all top-level keys as queryable fields. @timestamp, @message, @logStream, @log are always available.
EKS control plane logs: you can query audit logs with Insights to investigate who called what API when — useful during incident response. Runs on IA groups too.
06 Live Tail
What it does
Streams log events in near-real-time directly in the AWS console — like tail -f for CloudWatch. Supports pattern filtering to reduce noise. Launched 2023.
Limitation
Standard log groups only. IA log groups don't support it. This is why you verified no one was using Live Tail on the 8 EKS groups before converting them to IA.
Cost
$0.01 per GB of scanned log data while a Live Tail session is active. Close the session when done.
07 CloudWatch Metrics
Namespace
Container for metrics from the same service. AWS services use AWS/EC2, AWS/EKS, AWS/RDS, etc. Your custom metrics go in a namespace you choose: e.g. Blinq/App.
Dimensions
Key-value pairs that slice a metric into sub-groups. CPUUtilization in namespace AWS/EC2 with dimension InstanceId=i-abc123 is a different metric from the same name with a different instance ID.
Statistics
Average, Sum, Min, Max, SampleCount, pNN.NN (e.g. p99, p99.9). p-values require Extended Statistics and cost more. Average hides spikes — p99 is usually what you want for latency.
Resolution
Standard metrics: 1-minute granularity, stored for 15 months. High-resolution custom metrics: 1-second granularity, costs $0.30/metric/month extra. Use for latency-sensitive services.
Metric Math
Combine metrics with expressions in Grafana or CloudWatch dashboards. errors / requests * 100 → error rate %. FILL(m1, 0) → treat missing data as zero.
08 CloudWatch Alarms YOU BUILT THESE
States
OK — metric within threshold. ALARM — threshold breached for required evaluation periods. INSUFFICIENT_DATA — not enough data points yet (e.g. new alarm, metric stopped publishing).
Evaluation periods
"M of N consecutive periods must breach." Example: 3 of 5 one-minute periods above threshold → ALARM. Prevents single-spike false alarms. More periods = slower to alarm and recover.
Missing data treatment
breaching — treat as threshold exceeded (conservative). notBreaching — treat as OK (lenient). ignore — hold current state. missing → INSUFFICIENT_DATA. Default is missing.
Alarm actions
SNS topic (→ email, Slack, Lambda, PagerDuty), EC2 action (stop/start/reboot/recover), Auto Scaling action (scale out/in), Systems Manager OpsItem.
Threshold types
Static: fixed value. Anomaly detection: ML model learns normal band from historical data, alarms when metric exits the band. Useful for metrics with daily/weekly seasonality.
Composite alarms
Combine multiple alarms with AND/OR/NOT logic. Example: ALARM if (high-CPU AND high-memory) — reduces noise. Only fires actions on the composite state, not the individual alarms.
Your 6 GuardDuty-replacement alarms used CloudTrail metric filters → metrics → alarms → SNS. Pattern: CloudTrail event log → metric filter counts matching API calls → alarm on count > 0 → SNS → email.
09 Container Insights (EKS)
What it does
Deploys a CloudWatch agent as a DaemonSet on EKS nodes. Collects CPU, memory, disk, network metrics at the cluster, node, pod, and container level. Publishes to ContainerInsights namespace.
Cost consideration
Generates a large number of custom metrics ($0.30/metric/month). On a large cluster this adds up fast. Evaluate whether Prometheus + Grafana (which you implemented) gives better value.
Enhanced observability
Newer Enhanced mode also collects application signals and adds tracing. Requires additional CloudWatch agent config and costs more.
You chose Prometheus + Grafana over Container Insights at Blinq.IO. Know why: more control over metrics, lower cost at scale, richer dashboards, and integrates with your Alertmanager → Slack pipeline.

Karpenter provision flow
Pending pod → unschedulable event → NodePool evaluated → best-fit instance type → EC2 RunInstances → node joins → pod scheduled.
ArgoCD sync phases
PreSync hook → resource apply (by wave) → PostSync hook → health check. Sync waves order resources within the apply phase.
Kargo: Warehouse → Freight → Stage
Warehouse watches source. Freight = immutable snapshot. Stage = environment. Promotion = advance Freight into Stage with optional verification.
Prometheus rate() vs increase()
rate() = per-second average over window. increase() = total increase over window. Both extrapolate for partial intervals at range edges.
Terraform for_each vs count
for_each keyed by string → add/remove items without shifting indices and destroying unrelated resources. count is fragile for collections.
GitHub Actions OIDC to AWS
No stored keys. GitHub issues OIDC JWT → AWS STS AssumeRoleWithWebIdentity → short-lived creds. Needs IAM OIDC identity provider + role trust policy.
Zero-downtime deploy checklist
Readiness probe ✓ · maxUnavailable: 0 ✓ · PodDisruptionBudget ✓ · terminationGracePeriodSeconds ✓ · preStop hook if in-flight drain needed ✓
RDS Multi-AZ vs Read Replica
Multi-AZ = synchronous standby, automatic failover, HA. Read Replica = async copy for read scaling, manual promotion. Different purposes, different pricing.
IRSA flow
Annotate service account with IAM role ARN. EKS injects projected token. Pod exchanges token for AWS credentials via STS. No instance profile needed.
CloudFront ACM cert region
CloudFront cert MUST be in us-east-1, regardless of origin region. ALB certs live in the same region as the ALB.
